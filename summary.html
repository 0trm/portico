portico

a CLI that takes any input — text, code, URL, repo — and renders it as a structure: a three-layer ASCII visualization (roof / pillars / base).

what it does

  • Loads any input: raw text, file, directory, URL, or git repo (auto-detected).
  • Asks an LLM to decompose it into a fixed JSON shape — a roof (the unifying idea), 2–9 pillars (the load-bearing parts), and a base (the substrate).
  • Renders that JSON to ASCII. The renderer is pure: same JSON in, same bytes out — no LLM at draw time.

pipeline

loader → summarizer → analyzer → renderer

Strict boundary discipline. The summarizer fits oversized inputs to the model's context window. The analyzer's only job is JSON. The renderer takes validated JSON and produces ASCII — nothing else. The architectural metaphor never leaks into the schema; LLM calls never leak into the renderer.

design

  • Schema is the contract. Pydantic enforces 2–9 pillars, 1–4 base labels, and reasoning-first JSON fields (the model thinks before it labels — mitigates the format tax).
  • Abstention is first-class. Every output carries a fit_quality: good, stretched, forced, or not_applicable. Poems, flat lists, and recipes refuse — portico says no structure from this one and prints why instead of forcing pillars.
  • Labels live between two failure modes. Too generic ("Hypothesis", "Methodology") could apply to anything; too cryptic ("5→15→30→60") needs the input to decode. The prompt is calibrated to the middle, biased slightly abstract for clarity.
  • The renderer is testable without LLM calls. JSON fixtures → expected ASCII; byte-for-byte snapshots. 103 tests, ruff + pyright clean.

brand vocabulary

The mark _ii^ is itself the analogy: three characters, three layers. _ is the base, ii is the pillars, ^ is the roof. The glyphs are also working vocabulary in commits and prose.
